# The Clipboard Toolkit: 10 Coding Models, 12 Transforms

📋 All four benchmark prompts are on the **[Prompts page](prompts.html)**.

A second AHK v2 benchmark, built to fix a problem with the first reasoning test: that one was all-or-nothing, and a single trap zeroed everyone. This one is deliberately **broad and shallow** — twelve independent text transforms over the clipboard, each easy on its own. The spread comes from *how many* a model gets right, so models land across a gradient instead of pass/fail.

The task: a clipboard transform toolkit GUI exposing twelve `TF_*(text)` functions — five easy, four mid, three hard. Each is graded **functionally** against hidden inputs (feed text, call the function, assert the output), so "it runs" and "it's correct" are measured separately. A reference implementation scores 24/24, so the spec is provably solvable.

## Best Brain, Broken Hands

The top two tell the story. **Sonnet 4.6** gets 11 of 12 transforms *and* builds a working window — the cleanest all-rounder. **Opus 4.8** matches it on logic (11/12) but its **GUI crashes on launch** — it reached for `Gui.Edit.Pos`, a property that doesn't exist. Best evaluator in the field, hung on a hallucinated UI property. The functional and runtime signals disagree on purpose, and that disagreement is the most interesting cell in the table.

A textbook difficulty curve. The easy floor (`Upper`/`Lower`/`Title`) is 7/7 — everyone clears it. Then it thins: `SortLines` caught people out at 3/7 (case-insensitive *stable* sort is subtler than it looks), and the hard tail is where models pull apart. `Slugify` was surprisingly well-handled (6/7), `Base64` split the field (4/7), and **`NaturalSort` is the wall nobody cleared — 0/7.** Sorting `item2` before `item10` requires tokenizing digit runs and comparing them numerically; every model defaulted to lexicographic, putting `item10` before `item2`.

## Three Things the Run Surfaced

**The coding specialist face-planted again.** **Qwen3 Coder** managed just **3/12** — the same pattern from the GUI benchmark, where code-tuned models trail generalists on AHK's alpha syntax. A "coder" label is trained on mainstream languages; AHK v2.1 isn't one of them.

**Reasoning models hid their code in the wrong channel.** GLM-5, Kimi K2.6, MiniMax M3, and Gemini 3.1 Pro returned `content: null` with the actual program in the separate `reasoning` field — a one-shot-API quirk that initially looked like total failure until the harness read the right channel. Kimi went further and interleaved prose ("I'll define a Map of button labels…") directly into its code, which then won't parse.

**Three genuine parse failures.** GPT-5.1 (a 14 KB, complete file) died on `Missing ending "%"` — a botched dynamic dereference deep in the script. DeepSeek V4 Pro threw a plain syntax error. These weren't truncation; they're real AHK mistakes from strong models, the same one-shot fragility the GUI benchmark documented.

## Takeaway

This is the discrimination sweet spot the evaluator missed. By decomposing the task into a dozen independent, individually-easy, functionally-gradable features, the benchmark spreads ten frontier models across a real curve — from `Qwen3 Coder`'s 3/12 to `Sonnet 4.6`'s 11/12 — and the GUI-runs signal catches the model with great logic and a crashing window. Easy enough that everyone who parses scores something; hard enough at the tail (`NaturalSort`, `Base64`) that nobody runs the table.
