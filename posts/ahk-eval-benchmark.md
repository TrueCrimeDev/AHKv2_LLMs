# AHK-Eval: 36 Tasks, 181 Hidden Tests, 13 Models

All benchmark prompts are on the **[Prompts page](prompts.html)**.

**Round 2 is up:** [What a coding harness is actually worth](post.html?slug=ahk-eval-context-arms) — the same 36 tasks re-run with a rules-card arm and an execution-feedback arm across all thirteen models.

The [GUI benchmark](post.html?slug=llm-clipboard-benchmark) graded one big task. The [toolkit benchmark](post.html?slug=llm-toolkit-benchmark) graded twelve functions generated in a single shot — where one parse error zeroed a model's whole file. **AHK-Eval** is the third iteration, built the way standard code benchmarks (HumanEval, MBPP, LiveCodeBench) are built: **36 independent tasks**, each a single AHK v2 function, each generated by a separate one-shot API call at `temperature 0.2`, each graded functionally against **hidden test cases** the model never sees. A parse error now costs one task, not the run.

**How it's graded.** Every task ships a public example and 5-6 hidden cases (181 total). A driver wraps the submitted function, calls it with the hidden inputs, and string-compares the results. Three gates per task: does it *parse*, does it *run*, does it return the *right answer on every case*. A task is **solved** only at full marks; the `cases` column gives partial credit. Before any model ran, every task was verified solvable two independent ways — a Python oracle computed the expected outputs and a hand-written AHK reference matched them **181/181** against the same driver.

The tasks span six categories — strings, regex, data structures, numbers, dates, algorithms — in three difficulty tiers (12 easy, 12 mid, 12 hard).

## GPT-5.5 Was One Task From Perfect

**GPT-5.5 solved 35 of 36** — zero parse failures, 176 of 181 hidden cases. Its single miss: `NaturalSort`.

Which brings back an old wall. In the toolkit benchmark, NaturalSort (sort `item2` before `item10`) went **0-for-8**. Here, with a dedicated prompt and nothing else to build, it went **2-for-13** — eleven models still default to lexicographic comparison or fumble the digit-run tokenization. The two that cracked it: **Kimi K2.6** and **Fable 5**.

## The Kimi Redemption Arc

Kimi K2.6 is the strongest argument for per-task isolation. In the one-shot toolkit run it **failed to parse at all** — it interleaved its reasoning prose into the code. Here, generating one small function at a time, it scores **30/36 (162 cases — third-most in the field)** and is one of only two models to solve the hardest task. Same model, same temperature; the difference is granularity. Both numbers are real: it writes excellent AHK in small units and still has a packaging problem at file scale (its one parse failure here was `CsvField`).

## The Conflict-of-Interest Entry: Fable 5

By request, the model that built this benchmark also sat it. That needs guardrails, so here is exactly how it ran: **Claude Fable 5 over the bare Anthropic API** — one cold message per task, the identical prompt, no system prompt, no tools, and fresh contexts that never contained the reference solutions or hidden cases. One protocol deviation, disclosed: the Fable API rejects the `temperature` parameter outright, so it ran at default sampling where the others ran at 0.2. Grading used the same driver as everyone else.

Result: **34/36, second place** — 172 of 181 cases, zero parse failures, and one of the two NaturalSort solves. Its two misses are the most self-incriminating data in the suite, because both are AHK v2 semantics rather than logic:

- `ExtractDigits` — it wrote the field's favorite trap, `A_LoopField >= "0"`, and threw on the first letter. The same trap caught this benchmark's own reference implementations during authoring. Knowing about a trap in one context does not inoculate the next cold context.
- `ToRoman` — a textbook values/symbols loop that reads correct in any language and is wrong in this one:

```ahk
Loop values.Length {
    while (n >= values[A_Index]) {   ; while is a loop too — A_Index re-binds here
        result .= symbols[A_Index]
        n -= values[A_Index]
    }
}
```

In AHK v2, `while` maintains its **own** `A_Index`, shadowing the outer `Loop`'s. The algorithm is perfect; the index it reads is not.

The category cut repeats the toolkit's lesson at higher resolution: **data structures discriminate hardest**. `GroupByFirstLetter` and `Pivot` — both just "aggregate with a Map, sort the keys, join with the right separators" — went 3/12 each, worse than every algorithm task except NaturalSort. ## The One Trap That Ate the Field

The single most common failure in the suite isn't an algorithm — it's a **language semantic**. `ExtractDigits` ("return only the digit characters") is a one-liner: `RegExReplace(s, "\\D")`. Half the field instead wrote a parse loop with:

```ahk
if (A_LoopField >= "0" && A_LoopField <= "9")   ; throws TypeError on any letter
```

In AHK v2, relational operators on a non-numeric string **throw** — they don't compare alphabetically like v1, they don't coerce like JavaScript. The function dies on the first letter it meets, and an easy-tier task ends up with a 6/13 solve rate — failing GPT-5.1, Opus 4.8, Sonnet 4.6, Qwen3 Coder, Kimi K2.6, Mistral Large 3, and Fable 5 itself. (Full disclosure: the same trap caught this benchmark's own reference implementations twice before verification flushed it out. It is *the* AHK v2 trap.)

The GUI benchmark's headline failure mode also reappears at function scale: GPT-5.1's `WordWrap` logic is flawless and joins its lines with `StrJoin()` — a function AHK has never had. Parse-clean, hallucinated API, 0/5.

## What Changed From the Other Benchmarks

- **Parse failures nearly vanished.** The GUI benchmark rejected 35% of submissions at the loader; here parse failure is 2% (17 of 432). Small functions compile. The discrimination moved where it belongs: semantics.
- **The specialist pattern held for a third time.** Qwen3 Coder (16/36) trails every generalist again; Mistral Large 3 (9/36) sits last. Code-tuned training doesn't cover a niche v2 syntax.
- **The podium reshuffled.** Sonnet 4.6 won the toolkit; here Opus 4.8 leads it by four tasks, and Gemini 3.1 Pro — mid-pack on the toolkit with a crashing GUI — is second. Granularity changes rankings, which is exactly why one benchmark is never enough.

## Takeaway

AHK-Eval is the methodology the first two benchmarks were building toward: many small independent probes, hidden functional tests, partial credit, and a provably solvable reference. The result is the cleanest separation yet — 35/36 to 9/36 across thirteen frontier models — and the failures are no longer about getting a file to compile. They're about whether a model actually knows AHK v2: that relational operators throw on strings, that `StrJoin` doesn't exist, and that `item2` belongs before `item10`.

*Method notes: v2.1-alpha.30+Console fork for all parsing and execution; one API message per task via OpenRouter (Fable 5 via the Anthropic API), `temperature 0.2`, `max_tokens 8000`; fenced-code extraction with directive stripping; reasoning-channel fallback for models that return `content: null`. Harness, tasks, and grading driver were built and verified with Claude Fable 5; expected outputs come from a Python oracle independent of the AHK reference.*

### strings

**`AE_Acronym(s)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#7BC96F18;color:#7BC96F;vertical-align:middle">easy</span>

Return the acronym of the phrase: the first character of every space-separated word, uppercased, concatenated. Words are separated by one or more spaces.

*Example:* `AE_Acronym("for your information")` → `"FYI"`

**`AE_RepeatStr(s, n)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#7BC96F18;color:#7BC96F;vertical-align:middle">easy</span>

Return the string s repeated n times (n is an integer >= 0). n = 0 returns an empty string.

*Example:* `AE_RepeatStr("ab", 3)` → `"ababab"`

**`AE_SwapCase(s)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#F59E4218;color:#F59E42;vertical-align:middle">mid</span>

Return s with the case of every ASCII letter inverted (upper becomes lower, lower becomes upper). All other characters are unchanged.

*Example:* `AE_SwapCase("Hello World!")` → `"hELLO wORLD!"`

**`AE_WordWrap(s, w)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#F59E4218;color:#F59E42;vertical-align:middle">mid</span>

Greedy word wrap: s is a single line of words separated by single spaces. Pack words left to right; start a new line (\`n) when adding the next word (plus one joining space) would exceed width w. No word in the input is longer than w. Return the wrapped text joined with newlines (\`n), no trailing newline.

*Example:* `AE_WordWrap("the quick brown fox", 10)` → `"the quick
brown fox"`

**`AE_Caesar(s, k)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#DC354518;color:#DC3545;vertical-align:middle">hard</span>

Caesar cipher: shift every ASCII letter forward by k positions (k is an integer 0-25), wrapping around the alphabet, preserving case. Non-letters are unchanged.

*Example:* `AE_Caesar("abc XYZ", 3)` → `"def ABC"`

**`AE_ExpandTabs(s, n)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#DC354518;color:#DC3545;vertical-align:middle">hard</span>

Replace each tab (\`t) with spaces up to the next tab stop. Tab stops are every n columns (n >= 1). A tab always produces at least one space. Newlines (\`n) reset the column counter to 0. Return the expanded string.

*Example:* `AE_ExpandTabs("a	b", 4)` → `"a   b"`

### regex

**`AE_ExtractDigits(s)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#7BC96F18;color:#7BC96F;vertical-align:middle">easy</span>

Return only the digit characters of s, concatenated in order. If s has no digits return an empty string.

*Example:* `AE_ExtractDigits("a1b22c333")` → `"122333"`

**`AE_IsHexColor(s)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#7BC96F18;color:#7BC96F;vertical-align:middle">easy</span>

Return 1 if s is a valid CSS hex color — a '#' followed by exactly 3 or exactly 6 hex digits (0-9, a-f, A-F) — otherwise 0. Nothing else may precede or follow.

*Example:* `AE_IsHexColor("#1A2b3C")` → `1`

**`AE_ExtractEmails(s)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#F59E4218;color:#F59E42;vertical-align:middle">mid</span>

Extract every email address from s and return them joined by ';' in order of appearance. An email here is: one or more of [A-Za-z0-9._%+-], then '@', then one or more of [A-Za-z0-9.-], then a final '.' and 2+ ASCII letters. Return an empty string when there are none.

*Example:* `AE_ExtractEmails("mail bob@x.io now")` → `"bob@x.io"`

**`AE_StripTags(s)`** <span style="display:inline-block;font-size:10.5px;font-weight:600;padding:1px 7px;border-radius:5px;background:#F59E4218;color:#F59E42;vertical-align:middle">mid</span>

Remove every HTML-like tag from s: any substring starting with '<' and ending at the next '>'. There are no unmatched '<' in the input. Return the remaining text unchanged otherwise.

*Example:* `AE_StripTags("<b>hi</b>")` → `"hi"`
