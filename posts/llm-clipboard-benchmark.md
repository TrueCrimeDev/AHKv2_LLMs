# One Prompt, 77 Models: An AHK v2 Clipboard-Formatter Benchmark

I have a folder of AutoHotkey v2 scripts named after the model that wrote each one — `GPT-5-5-Pro.ahk`, `Gemini_3_Pro.ahk`, `Hermes_4_405B.ahk`, and six dozen more. Every file is the same model answering the **same prompt**. This post grades all of them on two escalating signals — does it *parse* and does it *run* — then digs into *why* the flagships diverge. **Every model name links to its own page** with full source, itemized scorecards, the captured window, and analysis.

All five benchmark prompts — primary, fallback, reasoning, toolkit, and the AHK-Eval template — are on the **[Prompts page](prompts.html)**.

## The Prompt

Every model received one strict spec: build a clipboard text-formatter as a single dependency-free AHK v2 GUI. The prompt is a full agent harness with a `P0_CRITICAL` rule block and an eight-item `DIAGNOSTIC_CHECKLIST`. The build target is specific and gradeable: load the clipboard on startup; three transforms (`StrUpper`/`StrLower`/`StrTitle`); multi-step undo + redo on `Ctrl+Z`/`Ctrl+Y`; a live char/line counter; a resizable, `Escape`-to-close window; and a `ControlWrapper` base class with two overriding subclasses.

## How Scoring Works

Two signals, both against the **v2.1-alpha.30+Console** fork.

**1. Parse (`/validate`)** — a parse-only load. Objective, but weak: it never runs a line. **2. Runs (launch + capture)** — each parsing script is launched, its window captured via `PrintWindow`, then killed. Builds a window → <span style="color:#7BC96F;font-weight:700">✓</span>, throws first → <span style="color:#DC3545;font-weight:700">✗</span>. Alongside runs, a **code score (0–100)** from the `DIAGNOSTIC_CHECKLIST`. The board sorts by **Runs, then code score** — a crash ranks below every program that opens.

## New on the Board: Claude Fable 5

Five new entries this update, all from one model: **Claude Fable 5**, Anthropic's newest flagship, answering the same prompt in five separate sessions under different harness configurations — from a bare no-skills session to a full knowledge-tree setup ([No Skills](model.html?id=Claude_Fable_5_NoSkills), [Skills](model.html?id=Claude_Fable_5_Skills), [AHK OOP](model.html?id=Claude_Fable_5_AHK_OOP), [Spotlight](model.html?id=Claude_Fable_5_Spotlight), and a [default config](model.html?id=Claude_Fable_5)).

The result is the cleanest sweep on the board: **all five parse, and all five run.** On a board where 35% of submissions don't compile and another quarter crash on launch, five-for-five at both gates is the deepest clean sweep on the board — nothing else fields more than two entries, and most families drop at least one of theirs. The sweep is what lifts Anthropic's family run rate to nine of ten.

The knowledge-tree entries also clear the bar almost nothing else does: [Fable 5 (Spotlight)](model.html?id=Claude_Fable_5_Spotlight) (rank 7, code 89, a 533-line build) and [Fable 5 (AHK OOP)](model.html?id=Claude_Fable_5_AHK_OOP) (rank 11, code 86) both ship **fully dark windows — 20/20 visual** — a feat previously exclusive to the two GPT-5.5 Pro entries. Where most of the field fights to produce *a* window, Fable 5's best entries produce a styled one.

The honest caveats: Fable's code scores peak at 89, below GPT-5.5 Pro's 95 — its one-shot windows are correct more than they are maximal — and the three plain-config entries render default-grey. And the same disclosure that applies to every `folder`-source entry applies here: these came from richer chat sessions, not cold API calls. (Full disclosure: Fable 5 also helped edit this post. The numbers above come from the same automated pipeline as everyone else's.)

## Clean Code That Doesn't Run

The runtime gate's sharpest lesson: *readability is not correctness.*

<div class="bm-wrap"><div class="bm-flow"><div class="bm-fb"><b>Hermes 4 405B</b>code score <b style="color:#fff">95</b> — tied best</div><div class="bm-arrow">→</div><div class="bm-fb bm-bad"><b style="color:#e0727d">✗ crashes</b>calls <code>IsMap()</code> — no such function → rank 31</div></div><div class="bm-flow" style="margin-top:8px"><div class="bm-fb"><b>GPT-5.5 Pro</b>code score <b style="color:#fff">95</b> — tied best</div><div class="bm-arrow">→</div><div class="bm-fb bm-good"><b style="color:#7BC96F">✓ runs</b>every symbol exists → rank 1</div></div></div></div>

Two identical code scores; opposite outcomes. Here is Hermes' base class — the most elegant `ControlWrapper` in the folder — and the one word that kills it:

```ahk
class ControlWrapper {
   Options := Map()
   __New(options := "") {
      this.Options := IsMap(options) ? options : Map()   ; IsMap() is not an AHK v2 function
   }
}
```

The fix is `options is Map`. It reads as expert AHK and throws `UnsetError` the instant `__New` runs. Contrast the winner, which treats transforms as data and uses only symbols that exist:

```ahk
this.transforms.Add("Upper", TextTransform("Upper", StrUpper))
this.transforms.Add("Lower", TextTransform("Lower", StrLower))
```

## The Class-as-Variable Trap

Before runtime, 26 outputs never parsed. The most common single error is the one the prompt explicitly warned about. AHK identifiers are **case-insensitive**, so `main` and `Main` are the same name:

```ahk
main := Main()        ; throws: Class cannot be used as an output variable
class Main {
}
```

[Phi-4](model.html?id=Phi-4) and [ERNIE 4.5](model.html?id=ERNIE_4-5_300B) both wrote exactly `main := Main()`. [GLM-4.6](model.html?id=GLM_4-6) reproduced the prompt's *verbatim invalid example*. The passers just renamed the variable — `editor := ClipboardEditor()`. The fix is one word; the warning wasn't enough to plant the reflex.

## The One-Shot Penalty

To stress-test with fresh models, I ran several live OpenRouter sweeps — 32 models, a dozen providers, one user message at `temperature 0.2`. The result is its own finding:

Hermes 4 (405B *and* 70B), Mistral Large 3, Llama 3.3 70B, Nemotron 3 Super, Opus 4.8 — all generated live, all broken. The folder's *runnable* entries are the pre-existing ones, pasted from richer chat sessions. The lesson isn't "these models are bad"; it's that **single-turn, low-temperature API generation of niche-syntax code is brittle** — context and iteration, not just model choice, decide whether it runs. (Both of Mistral's *coding-specialized* models, Codestral and Devstral, didn't even parse, while general models did better.)

## The Fallback Round

A second prompt — [`Prompt_Fallback.md`](https://github.com/TrueCrimeDev/AHKv2_LLMs) — exists for exactly this: when a model fails the hard spec, retry with a **stricter, more explicit, and simpler** brief. It hammers the exact failure modes (“never assume methods from other languages exist,” the class-as-variable trap, `Map()` has no `.Keys()`) and asks for a plainer three-button editor. I re-ran every reachable failure through it.

## Takeaways

- **Parse-rate is a vanity metric.** 65% parse; 41% run. A green `/validate` only says the syntax is plausible — launch generated code before you trust it.
- **Hallucinated APIs are the #1 runtime killer.** Eight of eighteen crashes are calls to functions or methods that don't exist — invisible to a linter and a parser alike.
- **Families have personalities.** OpenAI scores highest and finishes; Anthropic is the most reliable at running; Gemini writes clean code that crashes. Pick by the failure mode you can least afford.
- **One-shot API generation is brittle for niche syntax.** 2 of 30 live generations ran. Context and iteration beat raw model choice.
